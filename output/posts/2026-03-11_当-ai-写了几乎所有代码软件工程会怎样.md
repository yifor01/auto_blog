# 📌 【AI 寫 Code 的現實衝擊】當 AI 寫了幾乎所有程式碼，軟體工程會怎樣？

隨著 Opus 4.5、GPT-5.2 和 Gemini 3 等新模型的問世，軟體工程正在經歷一場悄然的革命。當 AI 寫了幾乎所有程式碼，開發者的角色會不會變得不再重要？這不是未來假想題，而是正在發生的事實。

## 🤔 **AI 寫 Code 的「頓悟時刻」已經來臨**

今年冬天，很多開發者在假期中嘗試用 AI Agent 完成個人專案，結果卻讓人驚訝。Google 首席工程師 Jaana Dogan 描述了她的體驗：「我不是在開玩笑，這一點都不好笑。去年我們在 Google 花了很長時間構建一個分布式智能體編排系統，我給 Claude Code 描述了一下這個問題，它一小時內就生成了我們去年才做出的東西。」

Vercel 的 CTO Malte Ubl 也分享了他的假期經歷：「我搞了兩個大型開源項目，一個還沒發布，另一個是用 TypeScript 給 AI 智能體做的完整 bash 環境實現。」

更魔幻的是，有人甚至在手機上寫出了生產級的軟體。透過 Claude Code for Web 連接 GitHub，在旅途中就能審查和合併包含新功能的 PR。

## ⚠️ **殘酷的現實：專業知識的價值在下降**

往前看，原型開發、精通多種程式語言、或者在某個技術棧上的專精，可能都會大幅貶值。當 AI 可以快速生成任何語言的程式碼，語言專精的門檻正在降低。

但這不是故事的結尾。當 AI 寫了幾乎所有程式碼，軟體「工程師」反而比以前更值錢。為什麼？因為技術負責人的特質更受追捧，初創公司會要求工程師具備產品思維，做一個扎實的軟體工程師而不仅仅是「寫程式碼的」。

## 💡 **令人不安的後果**

生成更多程式碼會帶來更多問題，薄弱的工程實踐會更快暴露，開發者的工作生活平衡可能更難維持。當 AI 可以快速產出大量程式碼，如何確保品質、維護性和安全性，變得更加重要。

## 🔄 **產品管理 vs 軟體工程：融合還是分離？**

產品經理現在可以更容易地生成軟體——實現目標所需的工程師更少了——但軟體工程師同樣不那麼需要產品經理了。兩個職業的重疊將比以往更多。

## 🎯 **生存策略：從「寫程式碼」到「工程思維」**

當 AI 寫了幾乎所有程式碼，開發者需要轉變的是：
- 從語言專精到系統設計能力
- 從程式碼實現到架構決策
- 從寫程式碼到定義問題
- 從執行者到引導者
