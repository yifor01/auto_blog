📌 【Q00/ouroboros】AI 寫 Code 仍靠「猜」？用規格鎖定意圖，終結非確定性編碼

你以為 AI 寫 Code 的瓶頸在模型能力？研究顯示，問題不在輸出，而在輸入：模糊的提示讓工程變成無限重來的猜測遊戲。

🤔 **工程瓶頸不在模型，而在人類的模糊表達**

當前 AI 編碼工具強大，但大多依賴 ad-hoc prompting：工程師試圖用語言「猜」出架構、邊界與約定，結果是反覆重寫、架構漂移，以及「看起來可用」卻無法驗證的半成品。這不僅降低生產力，更讓複製與除錯成本居高不下。

Ouroboros 提出 specification-first 的 Agent OS 架構，將非確定性 AI 編碼轉化為可重現、可觀察、policy-bound 的執行契約，試圖從根本上解決「輸入不清」的工程系統性風險。

🧪 **Agent OS 執行契約與多模型即插即用的本地運行時**

Ouroboros 是以本地為先的運行時層，設計為 Agent OS，透過結構化工作流程取代自由提示：
- interview → crystallize → execute → evaluate → evolve
- 將模糊構想轉化為不可變的規格種子（immutable seed spec），在寫程式之前先鎖定意圖
- 內建 3 階段自動化評估閘門，取代「看起來不錯」的人工驗證
- 作為 MCP 伺服器註冊，支援 Claude Code、Codex CLI、OpenCode 與 Hermes 的即插即用

安裝與啟動被壓縮為單一指令與自然語言對話，大幅降低進入門檻。

☑️ **用規格取代猜測，讓 AI 編碼可重現、可觀察、可進化**

- 架構漂移大幅減少：規格在寫程式前即被凍結，避免中途意圖改變
- 除錯與驗證成本降低：3 階段自動評估提供可觀察的品質閘門
- 多模型互操作實用化：同一規格可在不同 AI 編碼環境中重現執行
- 工作流程可重播：執行過程可追溯、可稽核，便於複製與持續演化

核心差異在於：不是讓 AI「寫更多」，而是讓規格「說清楚」，再讓執行「負責任」。

⚙️ **規格先行的工程權衡：靈活度換取確定性**

- 前期規格澄清需要更高的認知負擔，對快速原型習慣的工程師可能顯得繁瑣
- 流程依賴 MCP 相容工具鏈，非標準環境需額外整合
- 評估閘門仍依賴可觀測指標定義，無法取代所有層次的品質審查
- 長期演化與大型系統的可擴展性，仍需實戰驗證

這些限制並非缺陷，而是 specification-first 架構在靈活度與確定性之間的明確取捨。

🎯 **用 Socratic 訪談釐清需求，把 AI 當作驗證引擎而非代筆工具**

- 在下指令前，先用 interview 流程暴露隱含假設
- 將「幫我寫」替換為「解釋並約定規格」
- 保留可重播、可稽核的執行紀錄，讓進化有基線可依
- 結合本地運行時與 MCP 標準，提升跨工具的可攜性與復用性
