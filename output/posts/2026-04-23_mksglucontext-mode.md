### 🚀 **Context‑Mode 的三大核心設計**

| 功能 | 實作方式 | 效果 |
|------|----------|------|
| **Context Saving** | 透過沙箱工具將原始資料留在外部，僅把關鍵摘要寫入 SQLite。<br>例：315 KB 原始資料 → 5.4 KB 壓縮後 | **98% 資料減量** |
| **Session Continuity** | 每一次檔案編輯、git 操作、任務、錯誤與使用者決策都記錄於 SQLite，使用 FTS5 + BM25 召回相關事件。<br>壓縮時不把資料重新塞回 LLM，僅抽取必要片段。 | **會話可無縫續寫**，`--continue` 後即恢復完整上下文；不續寫則立刻清除，保證「乾淨」新會話。 |
| **Think in Code** | 讓 LLM 產生分析腳本，而非直接把大量檔案內容塞入上下文。<br>例如：寫一段程式統計函式數量，直接 `console.log()` 結果。 | **單腳本取代 10 次工具呼叫，節省 100 倍上下文**，符合所有 12 個平台的「LLM 為程式碼生成器」新範式。 |

### 📊 **核心發現：上下文壓縮不等於資訊遺失**  
- **原始 315 KB → 5.4 KB**（98% 減少）仍能在需要時即時檢索。  
- 透過 BM25 排序的關鍵事件檢索，使模型「只拿到」當前任務相關資訊，避免無關噪聲佔用上下文。  
- 完整的編輯與 git 歷史保存在本地 SQLite，除非使用者明確要求，永不被「壓縮」丟棄。

### 🔎 **深入分析：為什麼「思考程式碼」比「直接讀檔」更有效？**  
1. **資料密度提升**：一段簡短腳本可在執行時產出大量資訊，等同於把多個大型檔案的內容濃縮為單一結果。  
2. **降低上下文成本**：LLM 的 token 計費與上下文長度直接掛鉤，減少 token 消耗即等於降低推論成本。  
3. **提升可重用性**：腳本可以被快取、重複執行，未來相同分析只需呼叫一次，而非每次都重新塞入大量檔案。

### ⚠️ **研究限制**  
- **平台依賴**：目前實作以 SQLite + FTS5 為核心，對於不支援此資料庫的環境需自行改寫。  
- **首次索引成本**：將原始資料寫入 SQLite 仍需一次性 I/O，對極端低延遲需求的即時系統可能產生瓶頸。  
- **語義檢索精度**：BM25 以關鍵字為主，對高度抽象的程式語意仍可能錯過關聯性較低但重要的事件。

---

### 🎯 **實務啟示：如何在自己的 Agent 專案中導入 Context‑Mode**  
1. **將所有工具呼叫的原始輸出寫入 SQLite**，只把摘要或關鍵指標塞回 LLM。  
2. **使用 FTS5 + BM25** 於每次回應前檢索最近的編輯、git 操作與錯誤紀錄，僅把相關片段加入上下文。  
3. **把分析任務交給 LLM 產生腳本**，在 sandbox 環境執行並回傳結果，避免一次性讀入大量檔案。  
4. **在會話結束時決定是否保留**：`--continue` 續寫則保留 SQLite，否則直接刪除，確保新會話乾淨起點。
