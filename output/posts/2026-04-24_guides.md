📌 Simon Willison 指南：與 Claude Code 共寫程式，16 招避開「越寫越爛」陷阱

AI 寫 Code 的速度越快，程式庫裡留存的「不可解技術債」就越多；團隊發現，三個月前自動生成的模組，如今已經沒有人敢動。

🤔 **從原型到生產，Agentic Engineering 正在填補經驗空白**

當 Claude Code 與 OpenAI Codex 成為日常編程搭檔，開發者很快撞上同一個界線：一次性提示可以產出漂亮 Demo，但要如何讓 AI 在長週期、重構與協作場景中「不製造新問題」？Simon Willison 系統整理的可重複模式與反模式，正試圖回答這個從原型邁向生產的關鍵斷層。

🧪 **16 章實戰指南，將 Git、TDD 與 Subagents 織入日常流程**

本指南以 16 章結構化章節，涵蓋從基礊認知到具體工具鏈的完整路徑：
- 明確區分何謂 agentic engineering，以及「現在寫代碼很便宜」背後的思維轉向  
- 提出「囤積你已經會做的事」作為穩定判斷的基準  
- 示範如何結合 Git 分支策略、Subagents 分工、紅綠 TDD 迴圈與先行測試  
- 以實作案例（WebAssembly 與 Gifsicle 的 GIF 優化工具、博客轉訊息工具的新內容型別）展示完整工作流  
- 附帶可直接沿用的提示範本與避開常見反模式的檢查清單

核心設計特色在於：不將 AI 視為「黑盒外包」，而是將其嵌入已有的工程紀律中，透過流程約束降低協作與維護風險。

🟢 **用對流程的組，產出更乾淨的程式與更高的修改勇氣**

指南指出，當團隊採用「先跑測試 → 紅綠 TDD → Agentic 手動驗證 → 線性走查」的閉環時，AI 產出的程式在可讀性與可修改性上顯著提升；相較之下，缺乏測試與互動解釋的場景，則更容易累積難以追蹤的邊界狀態問題。

另一項關鍵觀察是：明確使用 Subagents 切割職責（例如：單獨負責測試生成、單獨負責邊界檢查），有助於減少單一提示過載所帶來的邏輯偏移，並讓 Code Review 更聚焦。

💡 **用 AI 協助理解與質量提升，而不是用 AI 取代思考負擔**

本指南強調的實務洞察可歸納為兩類模式：

- 高成效模式：以「先做測試 → 紅綠迴圈 → 交互式解釋」為主軸，AI 被用來澄清假設、產出可驗證的增量，並在 Git 歷史中保持可追溯的意圖  
- 低成效模式：將 AI 當作「整塊外包寫手」，缺乏測試先行與重構檢查，最終導致技術債加速堆疊

差異的核心不在模型能力，而在流程是否強制「理解可見」與「修改安全」。

⚠️ **以實務指南形式呈現，缺少大規模實證數據與長期跟蹤**

本指南的價值在於經驗匯整與可操作性，而非嚴格控制的實驗研究。因此其限制包括：未提供統計顯著的比較數據、樣本多為個人或小型團隊實踐、長期維護效果與組織擴展性仍需時間驗證。此外，所展示模式高度依賴對 Git、TDD 與現有測試基礎的熟練度。

🎯 **建議行動：把 AI 當作流程的延伸，而不是例外處理**

- 將「先跑測試」與「紅綠 TDD」設為 AI 介入的必經門檻  
- 用 Subagents 拆分校對、測試與重構職責，降低提示與邏輯負荷  
- 建立團隊級的「提示與反模式清單」，持續迭代何時該介入、何時該退回手動  
- 定期進行「交互式解釋」會議，確保 AI 產出的代碼可被整體消化
