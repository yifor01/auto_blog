📌 AI 應該有自己的電腦嗎？

你以為 AI 只會寫程式？Claude Cowork 證明它可以替你完成所有雜知識工作，甚至自己寫出自己。  
當技術團隊開始用編程助手來寫郵件、整理文件、規劃專案時，一個更深層的問題浮現：AI 的真正價值是否已經從「更好的聊天」轉移到「可信任的任務執行」？

🤔 AI 工具正從編程助手擴展到雜知識工作的主要引擎  
Latent Space 的訪談顯示，許多使用者將 Claude Code 用於各種雜知識任務，連原本專注於編程的技術建設者也把它當作非技術工作的得力助手。這種使用方式的廣度遠超出最初的設計預期，顯示出語言模型在知識密集型工作中的潛在需求。

🧪 透過人間 orchestration 多個 Claude Code 實例，Claude Cowork 在十天內原型成形  
Felix Rieseberg 與 Anthropic 團隊發現，只要讓幾位人類協調多個 Claude Code 執行實例，就能在短短一週半內讓工具具備完整的功能。這種「人間編排」的方式讓原型開發速度大幅提升，也驗證了內部既有組件可以快速被重組成新產品。

 用戶發現 Claude Code 主要被用於非編程工作，甚至技術團隊亦然  
實際觀察中，Claude Code 的使用場景遠不限於寫程式：撰寫報告、整理會議紀錄、產生知識庫條目等都成為常見用途。這意味著，即使是具備強大終端操作能力的工程師，也願意在更友好的介面上利用 AI 完成日常知識工作。

💡 執行成本下降使得『先建所有候選方案』成為可行策略，真正競爭點轉向可信任的任務執行  Felix 指出，隨著硬體與模型效率的提升，建構多個候選方案的成本已經變得足夠低，團隊可以「先把所有可能的解法都做出來」，再透過篩選與迭代挑選最佳方案。此時，產品的競爭重點不再是讓聊天更流暢或更有創意，而是如何讓 AI 能夠可靠地完成具體任務，而不需要人類不停地微調或監控。

⚠️ 目前僅為內部原型與訪談描述，缺乏公開基準測試與長期使用數據  
所有上述觀察皆來自於 Anthropic 內部的原型開發與 Felix 的訪談分享，尚未見公開的基準測試、使用者規模或長期追蹤研究。因此，關於 Claude Cowork 在真實生產環境中的穩定性、擴展性與長期使用效果，仍需後續驗證。

🎯 工程團隊可採用 local-first、VM 基礎的 agent 工作流，以提升非終端使用者的可用性而不犧牲擴展性  
Claude Cowork 採用 VM-based 架構，將原本只能在終端中運作的 Claude Code 包裝成更友好的桌面應用，使得不熟悉命令列的使用者也能獲得類似的 agent 能力。這種設計呼應了早期 VS Code 相對 Visual Studio 的簡潔外觀，但卻能隨著需求成長為高度可擴充與客製化的平台。對於想要在內部快速實驗 agent 工作流的團隊來說，這提供了一條低門檻且具備成長空間的路徑。
