📌 【技術逆向】揭開 gpt-oss 內建工具先驗機制

OpenAI 發表 gpt-oss-20b 時僅公開分數，未提供工具定義與 Agent 框架。社群原本認為無法驗證，研究卻發現：即使不給工具定義，模型照樣會以高置信度呼叫工具。這並非幻覺，而是深植權重的「強先驗」。

🤔 **官方未公開架構，開源社群面臨驗證斷點**

大型語言模型開源後，獨立複現 benchmark 分數一直是驗證模型真實實力的黃金標準。然而，OpenAI 在發布 gpt-oss-20b 時並未公開測試時使用的工具集與 Agent 調度架構（harness）。缺乏這些關鍵組件，社群無法確認官方分數是否依賴特定工程優化，也難以在自有環境中準確評估模型效能，形成開源透明度的驗證缺口。

🧪 **逆向推導訓練分佈，建構原生訊息格式測試框架**

為填補此缺口，研究者 Borislav Mavrin 採取兩階段逆向工程。首先，在不提供任何工具定義（tool definitions）的情況下對模型進行提示，觀察其輸出模式。結果顯示，模型仍會以極高的統計置信度呼叫符合其訓練分佈的工具指令，證實這是模型內化的行為特徵。第二步，團隊開發原生 Harmony Agent Harness，直接以模型訓練時使用的原生格式編碼訊息，繞過標準 Chat Completions API 常見的格式轉換損耗，重建完整的測試流水線。

📊 **成功獨立複現官方基準，分數誤差小於 0.4%**

透過上述設計，該研究達成首次獨立複現，且數據與官方報告高度吻合：
- SWE Verified HIGH：獨立 60.4%（官方 60.7%）
- SWE MEDIUM：獨立 53.3%（官方 53.2%）
- AIME25 (with tools)：獨立 91.7%（官方 90.4%）
三項關鍵基準的誤差均控制在 0.4% 以內，證實官方分數具有高度可重複性，並非依賴未公開的黑箱優化。

💡 **工具呼叫是內建肌肉記憶，原生格式避開轉換損耗**

核心洞察在於區分「動態提示」與「統計先驗」。一般開發者習慣在 prompt 中明確定義工具格式，但 gpt-oss-20b 在訓練階段已將特定工具呼叫模式深度融合至權重中，形成強先驗（strong prior）。這意味著模型對工具的熟悉度是內建的，而非依賴執行階段的 prompt 注入。此外，繞過 Chat Completions 的轉換層，直接採用原生訊息格式，能保留更完整的語意結構，避免系統層級的格式截斷或標記化（tokenization）偏差，這正是效能能精準對齊的關鍵工程決策。

⚠️ **聚焦單一尺寸模型，未驗證其他架構與長上下文**

本研究目前僅針對 gpt-oss-20b 單一尺寸進行驗證，未涵蓋該系列更小或更大參數量的變體。實驗設計專注於工具呼叫與程式解題基準，未測試長上下文視窗、多模態輸入或其他領域任務。此外，原生格式部署需依賴特定訊息編碼邏輯，若未來模型架構或 tokenization 策略調整，此 Harness 需同步更新方能維持相容性。

🎯 **部署 Agent 優先採用原生格式，降低轉換損耗**

- 在生產環境部署開源 Agent 時，若效能瓶頸來自 API 轉換層，可評估直接串接模型原生訊息格式，減少中間解析損耗。
- 針對已具備強工具先驗的模型，Prompt 工程可從「冗長定義格式」轉向「觸發內建行為」，簡化系統提示並降低 token 成本。
- 團隊已釋出 Harmony Agent Harness 原始碼，提供即插即用的參考實作，適合需要快速搭建高保真測試環境的工程團隊。
