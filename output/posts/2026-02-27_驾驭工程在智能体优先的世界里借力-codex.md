📌 【Codex 實驗室】0 行手寫程式碼，打造百萬行產品的 AI 工程團隊

你能想像一個軟體產品，完全由 AI 寫程式碼，卻照常上線、部署、修復漏洞嗎？我們團隊剛完成了這項實驗：過去五個月，我們用 Codex 從零開始構建了一個內部 Beta 產品，累積約百萬行程式碼，日活用戶數百人，而**人類從未手寫過任何一行程式碼**。

🤔 **AI 時代的工程師，該做什麼？**

當 Codex 能以 1/10 的時間完成原本需要人工數倍精力的開發工作，工程師的角色發生了根本性轉變。我們發現，未來工程師的主要工作不再是寫程式碼，而是：

- 設計系統環境與抽象層
- 明確表達開發意圖
- 建立反饋循環，讓 Codex Agents 可靠工作
- 構建智能體可讀、可執行的能力

🧪 **從空倉庫到百萬行程式碼的實驗**

2025 年 8 月，我們從一個空的 Git 倉庫開始。所有的腳手架——包括 CI 配置、格式化規則、包管理器設置——都由 Codex CLI 在 GPT-5 的驅動下生成。五個月後，倉庫累積約百萬行程式碼，涵蓋應用邏輯、基礎設施、工具鏈、文檔和內部開發實用程式。

關鍵數據：
- 3 名工程師處理並合併約 1,500 個 PR
- 每位工程師每天平均處理 3.5 個 PR
- 團隊擴充至 7 人後，吞吐量反而提升
- 數百名內部用戶正在使用該產品

 **重新定義的工程工作流程**

我們的核心信條是：**絕不手寫程式碼**。這迫使我們構建必要的體系，將工程速度提升數個量級。具體來說：

1. **深度優先策略**：將大目標拆解為更小的積木（設計、編碼、評審、測試等）
2. **智能體驅動**：通過提示詞引導 Codex 構建這些積木
3. **能力反饋**：當任務失敗時，反問「缺少什麼能力？如何讓這種能力對智能體既可讀又可執行？」

人類幾乎完全通過提示詞與系統交互：描述任務、運行智能體、讓它提交 PR。Codex 直接使用我們的標準開發工具（gh、本地腳本和倉庫內置技能）獲取上下文，無需人類在 CLI 中複製粘貼。

⚡ **讓系統「可讀」給 AI**

隨著程式碼吞吐量提升，瓶頸變成了人類 QA 的精力。我們致力於增強智能體對系統的感知能力，讓應用程式的 UI、日誌和指標能被 Codex 直接「讀懂」：

- 讓應用支持在每個 Git Worktree 中獨立啟動，Codex 能為每次變更啟動並驅動一個實例
- 將 Chrome DevTools 協議接入智能體運行時，創建處理 DOM 快照、截圖和導航的技能
- 構建臨時的可觀測性棧，對每個 Worktree 都獨立，Codex 可用 LogQL 查詢日誌

🎯 **對工程團隊的實戰啟示**

這項實驗揭示了 AI 時代工程團隊的幾個關鍵轉變：

1. **從編程到編程環境設計**：工程師的工作重心從寫程式碼轉向設計智能體工作的環境
2. **從個人開發到系統構建**：重點在於構建讓智能體可靠工作的體系，而非個人編程技巧
3. **從人工評審到智能體互評**：隨著時間推移，我們將絕大部分評審工作都推給了「智能體對智能體」的互評模式

💡 **未來已來，只是分布不均**

這不只是一個實驗，而是對未來軟體工程的預演。當 AI 能以 1/10 的時間完成開發工作，工程團隊的價值將從「寫得更快」轉向「設計得更好」、「系統構建得更可靠」。
