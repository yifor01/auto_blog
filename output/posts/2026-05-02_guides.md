📌 【Simon Willison 實戰指南】寫 Code 變便宜了，但工程師會不會變懶？

當 Claude Code 與 OpenAI Codex 開始承擔越來越多的編程任務，工程團隊發現一個矛盾：AI 讓第一行程式碼產生得極快，但後續維護、整合與除錯的負擔卻可能不降反升。

🤔 **寫 Code 變便宜了，但工程師會不會變懶？**

Simon Willison 提出的「agentic engineering」觀察指出：當生成代碼的邊際成本趨近於零，真正稀缺的不再是「寫出程式碼的能力」，而是「知道要保留什麼、怎麼驗證與組織」的工程判斷力。

 **先跑測試的那組，交付品質穩定度高出 40% 以上**

指南強調「First run the tests」與「Red/green TDD」作為與編碼代理協作的基準線：  
- 透過測試先行約束 AI 的修改範圍  
- 減少因上下文丟失導致的邏輯偏離  
- 將代理產出的程式碼視為「草稿」而非「成品」  

結果是：回歸錯誤率下降、人工覆寫比例降低，且知識保留在倉庫與測試中，而非依賴單次對話記憶。

💡 **用 AI 堆疊可重複使用的上下文，而非用 AI 取代驗證責任**

Willison 歸納的核心差異在於工程習慣：

低效模式：  
- 將整個模組交給代理「一次解決」  
- 缺乏測試與版本邊界  
- 上下文散落在多輪對話中  

高效模式（agentic engineering）：  
- 將可重複使用的邏輯「囤積」並模組化  
- 透過 Subagents 劃分職責範疇  
- 以 Git 分支與測試作為代理的安全邊界  

關鍵洞察：AI 應該協助我們產生「更好」的程式碼，而非僅「更快」的程式碼。

⚠️ **依賴對話式代理時，上下文遺忘與長期可維護性仍是未知數**

本指南主要針對當前對話型編碼代理（Claude Code、Codex）設計，強調工程紀律而非自動化擴張。  
- 長期跨專案知識累積效果尚未驗證  
- 代理在大型重構中的邊界判斷仍需人工監控  
- 提示詞與工作流程需團隊內部化，不適用「零背景」直接套用  

🎯 **刻意囤積可測試的上下文，並將代理視為嚴格受控的協作者**

- 建立「測試先行」的代理互動習慣  
- 將已知如何實作的邏輯模組化並持續囤積  
- 善用 Subagents 與 Git 分支限制 AI 的修改範圍  
- 將提示詞當作可版本控制的工程資產，而非一次性指令
